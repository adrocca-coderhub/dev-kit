# Jira Checklist Playbook

## Objetivo
Asegurar que una tarjeta Jira quede clara, robusta, entendible y útil tanto para negocio como para equipos técnicos.

---

## Antes de redactar la tarjeta
- [ ] entiendo el problema o necesidad
- [ ] tengo claro el objetivo
- [ ] conozco el contexto funcional o técnico
- [ ] sé qué flujo se impacta
- [ ] conozco al menos a alto nivel los archivos, tablas o componentes involucrados

---

## Contenido mínimo de la tarjeta Jira
- [ ] resumen
- [ ] objetivo
- [ ] contexto
- [ ] alcance
- [ ] flujo completo
- [ ] diagrama Mermaid
- [ ] funcionalidades implementadas
- [ ] datos utilizados
- [ ] input / output
- [ ] formas de ejecución
- [ ] casos representativos
- [ ] pendientes
- [ ] referencias o adjuntos
- [ ] resumen ejecutivo final
- [ ] criterios de aceptación

---

## Reglas de calidad
- [ ] explicar con palabras, no depender del código
- [ ] estructurar por secciones
- [ ] incluir tablas cuando ayuden a entender
- [ ] definir bien alcance y exclusiones
- [ ] incluir casos representativos
- [ ] dejar criterios de aceptación claros y verificables

---

## Datos y recursos
Verificar si la tarjeta necesita:
- [ ] archivos involucrados
- [ ] tablas o datasets
- [ ] columnas relevantes
- [ ] endpoints
- [ ] configuraciones
- [ ] documentos de referencia
- [ ] links adicionales

---

## Formas de ejecución
Si aplica, documentar:
- [ ] ejecución manual
- [ ] ejecución por script
- [ ] ejecución por scheduler o pipeline
- [ ] ejecución con parámetros distintos

---

## Criterios de aceptación
Los criterios deben permitir responder:
- [ ] ¿qué debe hacer exactamente?
- [ ] ¿qué inputs acepta?
- [ ] ¿qué output genera?
- [ ] ¿qué errores debe manejar?
- [ ] ¿qué evidencia demuestra que está completo?

---

## Revisión final
- [ ] alguien nuevo entendería la funcionalidad
- [ ] el flujo está claro
- [ ] el objetivo está bien definido
- [ ] los datos usados están documentados
- [ ] los criterios de aceptación son concretos
- [ ] no hay secciones vacías sin completar

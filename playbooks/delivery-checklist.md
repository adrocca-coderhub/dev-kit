# Delivery Checklist Playbook

## Objetivo
Tener una guía de cierre para entregar trabajo técnico de forma ordenada, clara y profesional.

---

## Antes de cerrar una tarea
- [ ] entendí completamente el requerimiento
- [ ] validé alcance y restricciones
- [ ] identifiqué dependencias o riesgos
- [ ] dejé documentación mínima necesaria
- [ ] actualicé artefactos relacionados si aplica

---

## Validación técnica mínima
- [ ] el cambio está ubicado en los archivos correctos
- [ ] la lógica es consistente con el flujo actual
- [ ] inputs y outputs están claros
- [ ] errores relevantes fueron contemplados
- [ ] no dejé código temporal innecesario
- [ ] el cambio mantiene coherencia con el proyecto

---

## Documentación
- [ ] PR documentada con el template correcto
- [ ] tarjeta Jira documentada si aplica
- [ ] Mermaid agregado si el flujo lo requiere
- [ ] se documentaron archivos impactados
- [ ] se documentaron datos, tablas o recursos usados
- [ ] se documentaron ejemplos relevantes

---

## Comunicación del cambio
- [ ] puedo explicar qué se hizo
- [ ] puedo explicar por qué se hizo
- [ ] puedo explicar el impacto
- [ ] puedo explicar riesgos o pendientes
- [ ] puedo explicar cómo se usa o ejecuta

---

## Entrega profesional
- [ ] la documentación está clara
- [ ] el contenido está ordenado
- [ ] no hay placeholders sin reemplazar
- [ ] no hay texto genérico innecesario
- [ ] el cambio se puede revisar sin demasiada fricción

---

## Cierre
Antes de dar por terminada la entrega, deberías poder resumir:
- objetivo
- cambio realizado
- flujo afectado
- archivos impactados
- datos usados
- output esperado
- pendientes o riesgos
